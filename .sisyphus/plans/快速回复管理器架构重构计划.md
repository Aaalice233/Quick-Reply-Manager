# 快速回复管理器架构重构计划

## TL;DR

> **目标**: 将 9,728 行的单文件屎山代码重构为多文件分层架构
>
> **方法**: 采用分层架构（Core → Utils → Services → Features → UI → Entry），最终打包为单个 JSON 脚本供酒馆使用
>
> **架构**: 状态管理使用模块级单例模式，UI 使用函数式组件，服务层封装副作用
>
> **交付物**: 约 27 个新文件，完全替代原有 index.ts
>
> **预计工作量**: 6 个并行波次，约 29 个任务（含主题系统拆分）

---

## Context

### 现状分析

- **文件规模**: `src/快速回复管理器/index.ts` 共 9,728 行
- **问题**: 单文件 IIFE 模式，职责混杂，可维护性差
- **技术栈**: TypeScript + Webpack + Tailwind CSS，可支持多文件打包

### 用户决策

- ✅ 完全重写（非渐进式重构）
- ✅ 多文件架构（采用分层设计）
- ✅ 最终保持单文件 JSON 输出（供酒馆导入）

### 约束条件

- 必须兼容 SillyTavern 酒馆助手无沙盒 iframe 环境
- 禁止使用 DOMContentLoaded，使用 `$(() => {})`
- 保持与原有数据结构兼容
- CSS 样式需内联到打包产物

---

## Work Objectives

### Core Objective

创建一个清晰的分层架构，将原本混杂在单文件中的代码按职责分离，提高可维护性、可测试性和性能。

### Concrete Deliverables

- 核心层: `types.ts`, `constants.ts`, `store.ts`
- 工具层: `utils/dom.ts`, `utils/data.ts`, `utils/validation.ts`, `utils/network.ts`
- 服务层: `services/storage.ts`, `services/llm.ts`, `services/placeholder.ts`, `services/debug.ts`, `services/theme.ts`,
  `services/theme.ts`
- UI 层: `ui/styles.ts`, `ui/components.ts`, `ui/workbench.ts`, `ui/preview.ts`, `ui/events.ts`
- 功能层: `features/categories.ts`, `features/items.ts`, `features/import-export.ts`, `features/settings.ts`
- 样式层: `styles/index.scss`, `styles/_tokens.scss`, `styles/_base.scss`, `styles/_components.scss`,
  `styles/_themes.scss`, `styles/_themes.scss`
- 入口层: 重写的 `index.ts`
- 文档更新: `AGENTS.md` 新增架构规则章节

### Definition of Done

- [ ] `pnpm build` 成功生成 `dist/快速回复管理器/index.js`
- [ ] 脚本在酒馆中正常运行
- [ ] 原有功能无回归
- [ ] AGENTS.md 已更新架构文档

### Must Have

- 所有类型定义从原文件提取到 types.ts
- 所有常量定义从原文件提取到 constants.ts
- CSS 样式从原文件提取到 SCSS 文件
- 状态管理逻辑从原文件提取到 store.ts
- 存储逻辑从原文件提取到 services/storage.ts
- UI 渲染逻辑从原文件提取到 ui/ 目录

### Must NOT Have (Guardrails)

- 不允许引入 Vue/React 组件框架（保持纯 TypeScript + DOM）
- 不允许引入 Pinia/Redux 状态管理库（保持简单单例模式）
- 不允许改变原有 CSS 类名和 CSS 变量名（保持向后兼容）
- 不允许改变数据存储格式（保持用户数据兼容）

  ## 快速回复管理器架构规范

  ### 架构分层

  本项目采用分层架构，从上至下为：

  1. **Entry**: 入口文件 index.ts，负责初始化和组装
  2. **Features**: 业务功能模块 (categories, items, settings 等)
  3. **UI**: 用户界面层 (styles, components, workbench 等)
  4. **Services**: 业务服务层 (storage, llm, placeholder 等)
  5. **Utils**: 工具函数层 (纯函数，无副作用)
  6. **Core**: 核心定义层 (types, constants, store)

  ### 导入顺序

  在 index.ts 中，按以下顺序导入模块：

  1. 类型定义: `import type { ... } from './types'`
  2. 常量: `import { ... } from './constants'`
  3. 状态: `import { ... } from './store'`
  4. 工具: `import { ... } from './utils/*'`
  5. 服务: `import { ... } from './services/*'`
  6. 功能: `import * as features from './features/*'`
  7. UI: `import { ... } from './ui/*'`
  8. 样式: `import cssContent from './styles/*.scss?raw'`

  ### 模块规范

  - 工具层 (utils/\*): 纯函数，无副作用，可独立测试
  - 服务层 (services/\*): 封装副作用，依赖下层
  - 功能层 (features/\*): 业务逻辑，依赖服务和工具
  - UI 层 (ui/\*): 渲染和交互，依赖所有下层

  ### 新增功能步骤

  1. 在 `features/` 创建新模块
  2. 如需新类型，添加到 `types.ts`
  3. 如需新常量，添加到 `constants.ts`
  4. 在 `index.ts` 导入并注册
  5. 更新本规范文档
